## Fix 1 (Critical): AgentBuilder 默认路径必须接入 context protocol tools

**File**: `src/agentos/builder.py` — `build()` method, lines 154-163

**Problem**: `AgentBuilder().provider(p).build()` produces an agent where:
- `ContextRenderer` tells the model to use `declare_schema`, `update_state`, `extend_schema`, `start_chapter`, `recall_context`
- But `ProviderRequestBuilder.tools` is empty (model can't call them)
- And `QueryLoop.tool_call_router` is None (calls can't be routed)

Result: default builder path creates an agent whose working-state protocol is broken.

**Fix**: In `build()`, when neither `.tools()` nor `.tool_call_router()` was called, still create a `ToolCallRouter` with empty `ToolRegistry` + `context_runtime`, and populate `provider_tools` from it.

Replace lines 154-163:

```python
tool_router = self._tool_call_router
provider_tools: list[ProviderToolSpec] = []

if self._tools is not None and self._tool_call_router is not None:
    raise ValueError(
        "AgentBuilder cannot use both .tools() and .tool_call_router(). "
        "Choose one tool setup.",
    )

if self._tools:
    tool_router = ToolCallRouter(
        tool_registry=tool_registry,
        context_runtime=context,
    )
    provider_tools = tool_router.tool_specs()
elif tool_router is not None:
    provider_tools = tool_router.tool_specs()
else:
    # Default path: context protocol tools must still be available
    tool_router = ToolCallRouter(
        tool_registry=tool_registry,
        context_runtime=context,
    )
    provider_tools = tool_router.tool_specs()
```

## Fix 4 (Critical): Anthropic adapter 必须输出 Anthropic 格式消息

**File**: `src/agentos/providers/anthropic.py` — `_message()` line 75-78

**Problem**: `_message()` delegates to `provider_message_to_dict()` which outputs OpenAI format. For `AssistantMessage` with tool_calls it produces `{"role": "assistant", "tool_calls": [...]}` — Anthropic expects `content: [{"type": "tool_use", ...}]`. For `ToolResultMessage` it produces `{"role": "tool"}` — Anthropic expects `{"role": "user", "content": [{"type": "tool_result", ...}]}`. Multi-turn tool-use conversations will 400.

**Fix**: Replace `_message()` with Anthropic-specific conversion:

```python
def _message(self, message: ProviderMessage) -> dict[str, object]:
    """Convert provider message to Anthropic Messages API format."""

    if isinstance(message, UserMessage):
        return {"role": "user", "content": message.content}

    if isinstance(message, AssistantMessage):
        content_blocks: list[dict[str, object]] = []
        if message.content:
            content_blocks.append({"type": "text", "text": message.content})
        for tc in message.tool_calls:
            content_blocks.append({
                "type": "tool_use",
                "id": tc.id,
                "name": tc.name,
                "input": tc.arguments,
            })
        return {
            "role": "assistant",
            "content": content_blocks if content_blocks else message.content,
        }

    if isinstance(message, ToolResultMessage):
        return {
            "role": "user",
            "content": [
                {
                    "type": "tool_result",
                    "tool_use_id": message.tool_call_id,
                    "content": message.content,
                },
            ],
        }

    return provider_message_to_dict(message)
```

**Important**: Check how `complete()` assembles the messages list. Anthropic requires strict user/assistant alternation. Multiple consecutive `ToolResultMessage` instances must be merged into a single `{"role": "user", "content": [...multiple tool_result blocks...]}` dict. If `complete()` just does `[self._message(m) for m in request.messages]`, add a post-processing step:

```python
def _merge_consecutive_tool_results(
    self, messages: list[dict[str, object]]
) -> list[dict[str, object]]:
    """Merge consecutive user/tool_result messages for Anthropic alternation rule."""
    merged: list[dict[str, object]] = []
    for msg in messages:
        if (
            msg["role"] == "user"
            and isinstance(msg["content"], list)
            and merged
            and merged[-1]["role"] == "user"
            and isinstance(merged[-1]["content"], list)
        ):
            merged[-1]["content"].extend(msg["content"])  # type: ignore[union-attr]
        else:
            merged.append(msg)
    return merged
```

## Fix 5 (Important): async cancellation 应中断后台 provider 调用

**File**: `src/agentos/runtime/agent.py` — `_stream_sync_in_executor()` lines 166-199

**Problem**: On `CancelledError`, code calls `future.cancel()` + `query_loop.request_interrupt()`. But `future.cancel()` cannot stop a thread already running in ThreadPoolExecutor. If the provider is mid-HTTP-stream, the worker continues running until the provider response completes. Meanwhile the session may be released and the agent left in interrupted state affecting the next request.

**Fix**: The worker must check the interrupt flag between iterations. The current `QueryLoop.run_turn_stream()` already checks `_interrupted` between tool iterations. The gap is during a single long provider streaming call. Add a cooperation mechanism:

1. In the worker function, after each `queue.put(event)`, check if the future was cancelled:

```python
def worker() -> None:
    try:
        for event in factory():
            put_future = asyncio.run_coroutine_threadsafe(queue.put(event), loop)
            put_future.result()
            if future_ref.cancelled():
                break
    except BaseException as error:
        asyncio.run_coroutine_threadsafe(queue.put(error), loop).result()
    finally:
        asyncio.run_coroutine_threadsafe(queue.put(None), loop).result()
```

2. Store a reference to the future before starting it:

```python
future_ref: asyncio.Future[None] | None = None

def worker() -> None:
    ...check future_ref.cancelled()...

future_ref = loop.run_in_executor(None, worker)
future = future_ref
```

This ensures the worker stops at the next event boundary (between stream chunks). It won't stop mid-HTTP-read, but it will stop before processing the next chunk, which is the best possible without cancelling the HTTP connection itself.

3. Also ensure `request_interrupt()` is always called on cancellation (already done in `async_stream` but not in `_stream_sync_in_executor` directly).

**Also**: Deduplicate `Agent._stream_sync_in_executor` and `AsyncQueryLoop._iterate_in_executor` — they are identical. Extract to a shared utility in `src/agentos/runtime/_async_bridge.py`:

```python
async def iterate_sync_in_executor(
    factory: Callable[[], Iterator[T]],
) -> AsyncIterator[T]:
    """Bridge a synchronous iterator to async via thread executor."""
    ...
```

Both `Agent.async_stream` and `AsyncQueryLoop.run_turn_stream` should delegate to it.

## Fix 7 (Important): HookManager 必须拒绝 async handler

**File**: `src/agentos/hooks/manager.py` — `dispatch()` line 36

**Problem**: If a user registers an `async def` handler, calling it returns a coroutine object (not None, not a HookResult). The coroutine is truthy, so `if result is None or result.action == "allow"` fails on `.action` attribute access — or worse, if it somehow passes, the coroutine is never awaited and silently dropped.

**Fix**: In `HookRegistry.register()` (or in `HookManager.dispatch()`), detect async handlers and raise immediately:

```python
# In hooks/registry.py register() method, add at the top:
import asyncio
if asyncio.iscoroutinefunction(handler):
    raise TypeError(
        f"async handler not supported in synchronous HookManager; "
        f"got {handler!r}. Use a synchronous function."
    )
```

**Test** (add to `tests/hooks/test_runtime.py`):

## Fix 8 (Minor): ProviderToolCall.arguments 缺少 deepcopy

**File**: `src/agentos/providers/base.py` — `ProviderToolCall` dataclass, line 17-23

**Problem**: `arguments: dict[str, object]` field is not deepcopied on construction. The frozen dataclass prevents reassignment of `self.arguments`, but callers holding a reference to the original dict can mutate it through the reference, breaking immutability.

**Fix**: Add `__post_init__` with deepcopy (same pattern as `ProviderFunctionSpec`):

```python
@dataclass(frozen=True, slots=True)
class ProviderToolCall:
    """provider 返回的标准化工具调用。"""

    id: str
    name: str
    arguments: dict[str, object] = field(default_factory=dict)

    def __post_init__(self) -> None:
        object.__setattr__(self, "arguments", deepcopy(self.arguments))
```

Add `from copy import deepcopy` to imports if not already present.

**Verify**: `uv run pytest -q` (no new test needed — existing tests cover construction)

## Fix 9 (Minor): ProviderResponse.tool_calls 应该是 tuple

**File**: `src/agentos/providers/base.py` — `ProviderResponse` line 67

**Problem**: `tool_calls: list[ProviderToolCall]` is mutable inside a frozen dataclass. External code can `response.tool_calls.append(...)`.

**Fix**: Change to tuple and normalize in `__post_init__`:

```python
@dataclass(frozen=True, slots=True)
class ProviderResponse:
    """provider 返回的标准化响应。"""

    content: str = ""
    tool_calls: tuple[ProviderToolCall, ...] = ()
    stop_reason: str | None = None
    usage: ProviderUsage | None = None
    model: str | None = None
    provider_name: str | None = None
    response_id: str | None = None
    thinking_content: str | None = None

    def __post_init__(self) -> None:
        if isinstance(self.tool_calls, list):
            object.__setattr__(self, "tool_calls", tuple(self.tool_calls))
```

**Breaking change check**: grep all callers that construct `ProviderResponse` — they currently pass `list`. The `__post_init__` normalization handles this. Grep callers that `.append()` to `tool_calls` — those must be fixed.
